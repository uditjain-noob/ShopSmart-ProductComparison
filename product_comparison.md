# Product Comparison Tool — Technical Documentation

A Chrome browser extension + Python backend that scrapes Amazon product pages, generates AI-powered product profiles, produces a structured side-by-side comparison, and serves a personalised recommendation via a questionnaire — all triggered on demand, with no persistent storage.

## System Overview

The tool has two independently deployable parts:

| Part | Technology | Role |
|---|---|---|
| Chrome Extension | Manifest V3, vanilla JS | Product collection, list management, triggering comparisons, rendering results |
| Python Backend | FastAPI + Uvicorn | Scraping, AI profiling, comparison generation, questionnaire |

Communication is via a local HTTP API on `localhost:8000`. The extension never reads product pages directly — all scraping and AI work is delegated to the backend.

## User Flow

```
Browse to Amazon product page
        ↓
Click extension icon → Add Product tab
        → "Add Current Page" or paste URL
        → Assign to a named list
        ↓
Repeat for 1–4 more products (min 2, max 5 per comparison)
        ↓
My Lists tab → tick products to compare (≥ 2)
        ↓
Click "View Comparison (N selected)"
        ↓
Extension POSTs URLs to POST /compare → job_id returned
Results tab opens immediately
        ↓
Results tab polls GET /compare/{job_id} every 3 s
Live progress shown:
  ● Scraping product data
  ● Generating product profiles
  ● Writing comparison, recommendation & questionnaire
        ↓
On completion, results rendered (in order):
  0. Skipped-products warning (if any)
  1. Personalisation questionnaire (5 questions)
  2. Comparison Summary
  3. Recommendation
  4. Pros & Cons cards (one per product)
  5. Specifications table
        ↓
User answers questionnaire → POST /compare/{job_id}/recommend
Best-match product highlighted with badge + banner
        ↓
"Save as PDF" → window.print() → browser print dialog
```

## Extension UI

### Popup — Add Product Tab

- Detects whether the current page is a supported Amazon product URL via `content.js`
- Displays the detected page title and a one-click **"Add Current Page"** button
- Manual URL paste input for adding products not on the current tab
- "Add to list" dropdown — creates a new list if none exist
- All state persists in `chrome.storage.session` (cleared on browser close)

### Popup — My Lists Tab

Each list is rendered as a collapsible card.

| Control | Behaviour |
|---|---|
| List header | Shows name + product count |
| Rename (✏) | Inline rename, saved to session storage |
| Delete (🗑) | Removes list and all its products |
| Select all / Deselect all | Toggles all product checkboxes in the list |
| Product checkbox | Includes/excludes from next comparison |
| Remove (×) | Deletes one product from the list |

Footer shows **"View Comparison (N selected)"** — enabled only when ≥ 2 products are checked across any combination of lists.

### List rules
- Max **5 products** per list
- Unlimited named lists
- **Session-scoped** — all lists are cleared when Chrome is closed (`chrome.storage.session`)

### Theme toggle
A sun/moon icon in the popup header switches between dark and light mode. Preference is persisted in `chrome.storage.local` (survives browser restarts).

## Results Page

Opens in a new tab (`results.html`). Immediately starts polling the backend.

### Loading state
Three animated step indicators with states: `pending → active → done`:
1. Scraping product data
2. Generating product profiles
3. Writing comparison, recommendation & questionnaire

### Skipped-products warning
If any product URLs fail after all retry attempts, an amber warning banner appears at the top of the results page listing each skipped product and its failure reason. The comparison proceeds with the products that succeeded (as long as ≥ 2 remain).

### Sections (in display order)

#### 0. Find Your Best Match (Questionnaire)
Five multiple-choice questions generated by the AI, specific to the products being compared. See [Personalised Questionnaire](#personalised-questionnaire).

#### 1. Comparison Summary
3–5 paragraph AI-written narrative covering key differences, trade-offs, and market positioning of the compared products.

#### 2. Recommendation
2–3 paragraph conclusion with a clear winner or use-case-based guidance, backed by the scraped and profiled data.

#### 3. Pros & Cons (per product)
One card per product. Each card shows:
- Platform and price
- Sentiment score (`Very Positive | Positive | Mixed | Negative | Very Negative`)
- AI-distilled pros list
- AI-distilled cons list
- Up to 3 notable verbatim (or lightly edited) customer review quotes
- Best-match badge (`★ Best for You`) injected here after questionnaire submission

#### 4. Specifications
Side-by-side table. Only rows where ≥ 2 products have a value are shown — eliminates sparse single-product rows that add noise.

### Save as PDF
`window.print()` is called. The `@media print` stylesheet strips all interactive UI (buttons, spinner, questionnaire, best-match banner, navigation) and formats the content cleanly.

## Personalised Questionnaire

### Motivation
The comparison summary and recommendation are one-size-fits-all. The questionnaire personalises the recommendation by asking the user about their specific priorities.

### Generation
Questions are generated **in the same LLM call** as the comparison summary and recommendation (no extra API call). The comparator prompt instructs the model to produce exactly 5 questions that:
- Target **real, specific trade-offs** visible in the specs, pros, and cons of the compared products
- Are **not generic** (e.g. not "What is your budget?" if all products have the same price)
- Have exactly **4 concise answer options** (≤ 8 words each)
- Cover different dimensions: use case, priority trade-offs, budget sensitivity, workflow needs

Each question also has an **"Other"** free-text option so users can express an answer not in the predefined list.

### Flow

```
Comparison job completes
        ↓
result.questionnaire = { "questions": [...] } included in job response
        ↓
results.js renders questionnaire at top of page
        ↓
User answers all 5 questions → "Find My Best Match" button enables
        ↓
POST /compare/{job_id}/recommend  { "answers": { "q1": "...", ... } }
        ↓
Backend calls generate_personalized_recommendation (1 additional LLM call)
        ↓
Returns { "recommended_title": "...", "reasoning": "..." }
        ↓
Best-match banner shown, matching product card highlighted + badged
Questionnaire collapsed to a "done" confirmation strip
```

### LLM call count per comparison

| Step | Calls |
|---|---|
| Per-product profiling | N (one per product) |
| Comparison + questionnaire generation | 1 (merged into a single call) |
| Personalised recommendation (on demand) | 1 (only when user submits answers) |
| **Total** | **N + 1 baseline, N + 2 if questionnaire submitted** |

## Backend Pipeline

### 1. Scraping (`scraper.py` + `platforms/amazon.py`)

`scrape_product_with_enrichment(url)` identifies the correct platform via `get_platform_for_url()`, calls `platform.scrape_product(url)`, then attempts review enrichment from other supported platforms (currently a no-op with one platform).

### 2. Profiling (`profiler.py`)

Calls `call_llm()` with a structured prompt containing the raw product data. Returns a `ProductProfile` dataclass with `description_summary`, `pros`, `cons`, `sentiment_score`, and `notable_quotes`.

### 3. Comparison + Questionnaire (`comparator.py`)

Single `call_llm()` call producing a JSON object with three keys: `summary`, `recommendation`, and `questions` (the 5-question personalisation set). The Comparison dataclass is extended with a `questionnaire` field holding the questions.

### 4. Personalised Recommendation (`questionnaire.py`)

Called on demand (not during the job). Takes the stored `ProductProfile` list, the questions from the comparison, and the user's answers. Returns `{ "recommended_title": str, "reasoning": str }` via a single LLM call.

### Per-product error isolation

Each scraping attempt and each profiling step is individually wrapped in `try/except`. Failures are recorded in `job["result"]["skipped_products"]` with descriptive messages. The job continues as long as ≥ 2 products remain viable. If fewer than 2 succeed at either stage, the job fails with a descriptive error.

## Scraping & Anti-blocking

### ScraperAPI integration (recommended)

When `SCRAPER_API_KEY` is set in `.env`, all Amazon requests are routed through ScraperAPI's proxy port method:

```
http://scraperapi.country_code={cc}:{API_KEY}@proxy-server.scraperapi.com:8001
```

- Country code is auto-detected from the Amazon domain (`amazon.in` → `in`, `amazon.com` → `us`, etc.)
- ScraperAPI handles IP rotation, CAPTCHA solving, and retries internally
- `verify=False` is required for ScraperAPI's SSL proxy (by design — their proxy terminates TLS)
- With ScraperAPI active, only 2 local retry attempts are made (ScraperAPI does the heavy lifting)

### Direct scraping fallback

Without `SCRAPER_API_KEY`, the scraper uses exponential back-off with ±25% jitter:

| Attempt | Base wait | Max wait (with jitter) |
|---|---|---|
| 1 | 0 s (immediate) | — |
| 2 | 3 s | 3.75 s |
| 3 | 6 s | 7.5 s |
| 4 | 12 s | 15 s |
| 5 | 24 s | 30 s |
| 6 | 48 s | 60 s |
| 7 | 96 s | 120 s |

- User-Agent rotates across 5 real browser strings on each attempt
- Non-retryable HTTP errors (404, 410, etc.) fail immediately without retrying
- CAPTCHA detection checks for known Amazon bot-challenge phrases in the page text
- Exhausting all attempts raises a `RuntimeError` with a clear message directing the user to configure ScraperAPI

### Inter-request delay

A 3-second sleep is inserted between each product scrape to reduce the chance of rapid-request rate limiting even when ScraperAPI is active.

### Logging

All scraping activity is logged at `INFO` / `WARNING` / `ERROR` level via Python's `logging` module:

```
10:42:01  INFO   ...amazon  [Amazon] ScraperAPI proxy active (country=in)
10:42:01  INFO   ...amazon  [Amazon] Attempt 1 — GET https://www.amazon.in/...
10:42:04  INFO   ...amazon  [Amazon] Attempt 1 — HTTP 200
```

## AI / LLM Layer

### Provider

**Google Gemini** via the `google-genai` SDK.

| Setting | Default | Override |
|---|---|---|
| Model | `gemini-2.5-flash` | `GEMINI_MODEL` in `.env` |
| API key | — | `GEMINI_API_KEY` in `.env` (required) |
| Temperature | 0.2 | Hardcoded |
| Max tokens | 4096 | Hardcoded |

Available models on the key can be enumerated with:
```python
from google import genai
client = genai.Client(api_key="...")
print([m.name for m in client.models.list()])
```

### Robust JSON parsing (`llm.py → parse_llm_json`)

All LLM responses that must be JSON are processed through `parse_llm_json()` rather than a raw `json.loads()`. It handles the three most common model output failures:

1. **Markdown code fences** — strips ` ```json ... ``` ` wrappers
2. **Bare newlines inside string values** — invalid in JSON but common when models write multi-paragraph text; a character-level state machine walks the string, tracking quote/escape state, and replaces `\n`, `\r`, `\t` with their JSON escape sequences only when inside a string value
3. **Stray text** — extracts the outermost `{…}` block before attempting to parse

Fast path: tries `json.loads()` directly first; state-machine repair only runs if that fails.

## Theming

Both the popup and results page implement a **"Sage & Slate"** colour palette via CSS custom properties.

| Theme | Activation |
|---|---|
| Dark (default) | `:root` — no attribute required |
| Light | `[data-theme="light"]` on `<html>` |

Switching is done by `toggleTheme()` in `popup.js` and `results.js`, which adds/removes the `data-theme="light"` attribute on `document.documentElement`. Preference is saved to `chrome.storage.local` and restored on page load via `initTheme()`.

### Key CSS variables

```css
--primary        /* Sage green — primary actions, borders, selected states */
--primary-light  /* Translucent sage — hover backgrounds */
--peach          /* Gold/peach — questionnaire, best-match accents */
--card           /* Card background */
--bg             /* Page background */
--text           /* Primary text */
--text-muted     /* Secondary/placeholder text */
--border         /* Border colour */
--success        /* Done/confirmed states */
```

## How to Run

### Prerequisites
- Python 3.11+
- [uv](https://docs.astral.sh/uv/) installed globally
- Gemini API key
- Google Chrome

### 1. Install dependencies

```bash
cd Assignments/Assignment_1
uv sync
```

### 2. Configure `.env`

```dotenv
GEMINI_API_KEY=your_key_here
SCRAPER_API_KEY=your_scraperapi_key_here   # optional but recommended
```

### 3. Start the backend

```bash
uv run uvicorn backend.api:app --reload --port 8000
```

### 4. Load the Chrome extension

1. `chrome://extensions` → enable Developer mode
2. **Load unpacked** → select `Assignments/Assignment_1/extension/`

### 5. CLI mode (no extension)

```bash
uv run main.py
```

Interactive terminal UI using the same pipeline.

## API Reference

### `POST /compare`
Start a new comparison job.

**Request body:**
```json
{ "urls": ["https://amazon.in/...", "https://amazon.in/..."] }
```
- Minimum 2 URLs, maximum 5
- All URLs must be from a supported platform

**Response:**
```json
{ "job_id": "uuid-string" }
```

---

### `GET /compare/{job_id}`
Poll job status.

**Response (running):**
```json
{
  "status": "running",
  "progress": "Scraping product 2 of 3…",
  "result": null,
  "error": null
}
```

**Response (complete):**
```json
{
  "status": "complete",
  "progress": "Done",
  "result": {
    "products": [ { "title": "...", "price": "...", "pros": [...], ... } ],
    "summary": "...",
    "recommendation": "...",
    "questionnaire": { "questions": [ { "id": "q1", "text": "...", "options": ["A","B","C","D"] } ] },
    "skipped_products": [],
    "markdown": "..."
  }
}
```

**Response (error):**
```json
{ "status": "error", "error": "Could not scrape…", "progress": "Failed" }
```

---

### `POST /compare/{job_id}/recommend`
Get personalised recommendation based on questionnaire answers. Only valid after job is `complete`.

**Request body:**
```json
{ "answers": { "q1": "Heavy video editing", "q2": "Battery life", ... } }
```

**Response:**
```json
{
  "recommended_title": "Apple 2026 MacBook Pro…",
  "reasoning": "Your emphasis on video editing and willingness to trade battery life for performance aligns strongly with the MacBook Pro's M4 Pro chip…"
}
```

---

### `GET /platforms`
List all registered platforms.

**Response:**
```json
{ "platforms": [ { "name": "Amazon", "base_url": "https://www.amazon.com" } ] }
```

---

### `GET /health`
```json
{ "status": "ok" }
```

## Data Models

```python
@dataclass
class Review:
    rating: float
    title: str
    body: str
    source: str          # platform name

@dataclass
class ProductData:
    url: str
    platform: str
    title: str
    price: Optional[str]
    description: str
    specs: dict[str, str]
    reviews: list[Review]
    rating: Optional[float]
    rating_count: Optional[str]

@dataclass
class ProductProfile:
    title: str
    price: Optional[str]
    platform: str
    specs: dict[str, str]
    description_summary: str
    pros: list[str]
    cons: list[str]
    sentiment_score: str   # "Very Positive" | "Positive" | "Mixed" | "Negative" | "Very Negative"
    notable_quotes: list[str]

@dataclass
class Comparison:
    products: list[ProductProfile]
    summary: str
    recommendation: str
    markdown: str
    questionnaire: dict   # { "questions": [ { "id", "text", "options": [4 items] } × 5 ] }
```

### Adding a new platform

1. Create `backend/platforms/{name}.py` subclassing `BasePlatform`
2. Implement: `name`, `base_url`, `can_handle(url)`, `scrape_product(url)`
3. Append an instance to `SUPPORTED_PLATFORMS` in `backend/platforms/__init__.py`

Everything else (scraping orchestration, review enrichment, comparison, the extension) picks it up automatically.
